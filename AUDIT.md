# Vexart Full-Project Audit Report

**Date:** 2026-04-26
**Auditor:** Gentleman AI (Senior Architect)
**Scope:** Full codebase — all packages, native library, build config, tests
**Verdict:** **120+ issues found across 4 parallel audit streams. Multiple CRITICAL bugs including undefined behavior in Rust, broken virtualization, non-reactive router, non-interactive styled Button, unrecoverable FFI error paths, and zero test coverage on 3 of 6 packages.**

---

## Executive Summary

This codebase was generated by GPT Codex. It shows a recognizable pattern: **structurally impressive scaffolding with deeply flawed implementations**. The architecture documents are excellent, the FFI boundary design is sound in intent, but the actual code has:

1. **Undefined behavior in Rust** — raw pointer aliasing that violates Rust's safety model in 11+ places
2. **Mutex poisoning** that permanently bricks the renderer after any GPU error
3. **A "VirtualList" that doesn't virtualize** — renders from index 0 on every frame
4. **`RouteOutlet` is NOT reactive** — navigation visually does nothing after `router.push()`
5. **Styled `Button` is non-interactive** — no `onPress`, no `focusable`, a dead box
6. **`className` theming is static** — reads `colors` at module load, ignores `setTheme()`
7. **Zero test coverage** on `@vexart/headless` (28 components), `@vexart/styled`, and `@vexart/primitives`
8. **Silent FFI error swallowing** — most native calls don't check return codes
9. **SHM resource leaks** — shared memory segments accumulate in `/dev/shm` indefinitely
10. **`useFocus` never unregisters** — Tab cycles through ghost entries from unmounted components
11. **Render loop dies silently on exception** — no catch in frame timer, loop never re-schedules
12. **`setTimeout(tick, 0)` for animations** — CPU-bound tight loop that monopolizes event loop
13. **Broken headless components** — DialogClose does nothing, Tooltip ignores placement, Textarea can't scroll, Select compound mode is non-functional

The project needs a focused stabilization effort before any feature work.

## 1. CRITICAL Issues

### CRIT-01: Undefined Behavior — Raw Pointer Split-Borrow in Rust (11+ locations)

**Package:** `native/libvexart`
**Files:** `paint/mod.rs:229-340`, `composite/mod.rs` (6 locations), `text/mod.rs:429-551`

The WGPU dispatch code creates multiple aliasing references to `self` via raw pointer casts:

```rust
let rec_ptr: *mut TargetRecord = self.targets.get_mut(target).expect("...") as *mut _;
let view_ref: &wgpu::TextureView = unsafe { &(*rec_ptr).view };
let layer = unsafe { (*rec_ptr).active_layer.as_mut().expect("...") };
// Then also borrows &self.wgpu.device in the same scope
```

This violates Rust's aliasing rules. The compiler emits `noalias` on `&mut` references — LLVM is allowed to assume they don't alias. **This is undefined behavior** that currently works by luck but can miscompile on any rustc upgrade.

**Impact:** Silent data corruption, GPU crashes, "works in debug, crashes in release" scenarios.
**Fix:** Restructure dispatch to take `(&WgpuContext, &mut TargetRegistry)` as separate parameters so the compiler can prove disjointness.

### CRIT-02: Mutex Poisoning Permanently Bricks Renderer

**Package:** `native/libvexart`
**File:** `lib.rs` — 26 occurrences of `.lock().unwrap()`

Every FFI function does `SHARED_PAINT.lock().unwrap()`. If ANY panic poisons the mutex (and many can — see CRIT-03/04), every subsequent FFI call panics forever. The native renderer is permanently dead until process restart. Combined with the `ffi_guard!` catch, the user sees a blank terminal that never recovers.

**Impact:** A single transient GPU error = permanent renderer death for the Bun process lifetime.
**Fix:** Use `.lock().unwrap_or_else(|p| p.into_inner())` everywhere, or migrate to `parking_lot::Mutex` (no poisoning).

### CRIT-03: `expect()`/`unwrap()` in FFI-Reachable Render Paths

**Package:** `native/libvexart`
**Files:** `paint/mod.rs:250,258,463,472`, `composite/mod.rs` (12 locations), `text/mod.rs:465,472`

```rust
self.targets.get_mut(target).expect("target disappeared")
pctx.images.get(&image).unwrap()
device.poll(wgpu::PollType::Wait { ... }).expect("device poll failed")
```

These panic inside active WGPU RenderPasses. If the RenderPass `Drop` also panics (possible when encoder is in bad state), you get a double-panic = `SIGABRT` that kills the terminal.

**Impact:** A stale image handle from JavaScript can crash the entire process and corrupt terminal state.
**Fix:** Replace ALL `unwrap()`/`expect()` in FFI-reachable code with `match`/`else { return ERR_INVALID_HANDLE; }`.

### CRIT-04: GPU Context Creation Fails Silently, Returns 0n

**Package:** `@vexart/engine`
**File:** `packages/engine/src/ffi/gpu-renderer-backend.ts:879-880`

```typescript
const result = symbols.vexart_context_create(optsPtr, 0, ptr(ctxBuf)) as number
if (result !== 0) return 0n  // SILENTLY returns invalid handle
```

On GPU initialization failure, returns `0n` as the context handle. Every subsequent FFI call with `0n` either crashes or produces garbage. No error logging, no exception, no state tracking.

**Impact:** On any machine where WGPU fails (missing GPU, driver issue, CI), the entire renderer silently does nothing.
**Fix:** Throw `VexartNativeError` with the native error message from `vexartGetLastError()`.

### CRIT-05: `vexart_paint_dispatch` Never Checks Return Code

**Package:** `@vexart/engine`
**File:** `packages/engine/src/ffi/gpu-renderer-backend.ts:380`

```typescript
symbols.vexart_paint_dispatch(ctx, target, ptr(u8), total, ptr(_flushStatsBuf))
// Return code IGNORED
```

The most-called FFI function in the entire rendering pipeline doesn't check its error code. If GPU device is lost, shader fails, or buffer is corrupted, the code continues as if nothing happened.

**Impact:** GPU errors are completely invisible. Rendering corruption with no diagnostic path.
**Fix:** Check return code, call `vexartGetLastError()` on failure, and log/throw.

### CRIT-06: VirtualList Is Not Virtualized

**Package:** `@vexart/headless`
**File:** `packages/headless/src/collections/virtual-list.tsx:76`

```typescript
const visibleItems = () => props.items.slice(0, endIndex())  // Slices from 0!
```

The VirtualList slices from index 0 to endIndex — there is NO startIndex calculation and NO top spacer. With 100k items and scroll at the bottom, it renders ALL items from 0 to endIndex. The entire purpose of virtualization is defeated.

Additionally, the `<For>` callback uses `idx()` (slice-local index) for selection/highlight comparisons instead of absolute list index. Selection highlighting is broken when scrolled.

**Impact:** O(N) rendering instead of O(viewport). Performance death with large lists.
**Fix:**
```typescript
const startIndex = () => Math.max(0, Math.floor(scrollPos() / props.itemHeight) - overscan())
const visibleItems = () => props.items.slice(startIndex(), endIndex())
// Add top spacer: <box height={startIndex() * props.itemHeight} />
// Fix index: startIndex() + idx()
```

### CRIT-07: SHM Segment Leak — Never Unlinked

**Package:** `native/libvexart`
**File:** `kitty/shm.rs`, `kitty/transport.rs`

On every SHM frame, a `/vexart-kitty-{pid}-{image_id}` entry is created in `/dev/shm`. `shm_release(handle, 0)` only closes the fd — the segment is NEVER unlinked. Long-running processes silently exhaust shared memory space.

**Impact:** Memory leak proportional to session duration. Hours-long sessions can fill `/dev/shm`.
**Fix:** Track SHM names and unlink after the terminal has read them (short TTL queue).

### CRIT-08: Packed Buffer FFI Reads Without Length Validation

**Package:** `native/libvexart`
**File:** `lib.rs:761-890`

```rust
let bytes = std::slice::from_raw_parts(layer_ptr, 20); // No length param!
```

FFI functions accept raw pointers and read 12-20 bytes without any length parameter or validation. If the TypeScript side passes a short buffer, Rust reads past the end — undefined behavior.

**Impact:** Segfault or info disclosure from reading uninitialized memory.
**Fix:** Add a length parameter to all packed-buffer FFIs and validate before slicing.

### CRIT-11: Mouse Coordinates Stale After Terminal Resize

### CRIT-12: `RouteOutlet` Is NOT Reactive — Navigation Broken

**Package:** `@vexart/app`
**File:** `packages/app/src/router/router.tsx:234-254`

```typescript
export function RouteOutlet(props) {
  const match = contextRouter.match()  // Called ONCE at construction
  const Component = match.route.component
  let element = Component({ params: match.params })  // Invoked ONCE

```

`contextRouter.match()` reads signals at component construction, not inside a reactive scope. After `router.push("/foo")`, the rendered component does NOT change. **The entire app router is broken.**

**Impact:** Navigation visually does nothing. Users click links and nothing happens.
**Fix:** Use `<Show when={contextRouter.match()} keyed>` for reactive re-rendering.

### CRIT-16: `useFocus` Never Unregisters — Ghost Focus Entries

**Package:** `@vexart/engine`
**File:** `packages/engine/src/reconciler/focus.ts:139-144`


### CRIT-17: Render Loop Dies Silently on Exception

**Package:** `@vexart/engine`
**File:** `packages/engine/src/loop/loop.ts:393-431`

```typescript
timer = setTimeout(() => {
  if (isDirty()) frame()  // If frame() throws...
  scheduleNextFrame()      // ...this never runs
}, delay)
```

No try/catch around `frame()`. Any exception kills the render loop permanently — no more frames, blank screen.

**Impact:** A single rendering error freezes the entire application.
**Fix:** `try { frame() } catch (e) { console.error(e) } scheduleNextFrame()`.

### CRIT-18: `setTimeout(tick, 0)` Animation Driver — CPU Monopolization

**Package:** `@vexart/engine`
**File:** `packages/engine/src/loop/animation.ts:108-110`

```typescript
function tick() {
  if (!animating) return
  setValue(...)
  markDirty()
  timer = setTimeout(tick, 0)  // Tight loop!
}
```

Animation ticks use `setTimeout(0)` — a CPU-bound tight loop that monopolizes the event loop, blocking stdin I/O. Multiple concurrent animations multiply the problem.

**Fix:** Tie animations to the render loop's frame callback instead of independent timers.

### CRIT-19: `resetFocus` Leaks Input Handler — Double Tab Dispatch

`resetFocus()` sets `focusInputConnected = false` but never unsubscribes the existing `onInput` handler. After remount, `ensureFocusInput()` adds a SECOND subscriber. Tab fires twice.

### HIGH-07: Input/Textarea/Combobox Reject Multi-Byte Unicode Characters

**Files:** `input.tsx:170`, `textarea.tsx:544`, `combobox.tsx:108`

```typescript
if (e.char && e.char.length === 1 && ...) {  // Breaks with emoji, CJK
```

JavaScript `.length` counts UTF-16 code units, not characters. A single emoji like "😀" has `.length === 2`. All printable multi-byte characters are silently rejected.

**Fix:** Use `[...e.char].length === 1` or remove the length check entirely (check `e.char` existence is sufficient).

## Audit Methodology

This audit was conducted by 4 parallel review agents plus manual review:

1. **Engine Agent** — FFI bridge, reconciler, render loop, input, animation (30+ files)
2. **Headless/Primitives Agent** — All 28 headless components + 5 primitive wrappers

4. **Styled/App/Build Agent** — Theme, styled components, router, build scripts, tests, cross-cutting

Each agent was instructed to be adversarial and find bugs that could cause runtime failures.

## Key Pattern: GPT Codex Signature

The codebase exhibits a consistent pattern seen in AI-generated code:

1. **Architecturally impressive, runtime-broken** — The layer system design, FFI boundary model, and component API are well-structured. The implementations have fundamental correctness bugs.

2. **Wide surface area, thin testing** — 28 headless components, 22 styled components, 50+ FFI functions. Zero tests on 3 packages. The bugs above would be caught by a single integration test per feature.

3. **Comment-driven correctness** — Comments describe what the code SHOULD do, but the code doesn't do it. `DialogClose` says "closes the dialog when activated" but renders a fragment. `VirtualList` says "only renders visible items" but renders from index 0.

4. **Copy-paste divergence** — Layer boundary logic is duplicated between `walk-tree.ts` and `assign-layers.ts` with different module-level counters. `focus.ts` registration and `unregisterSubtree` have parallel but inconsistent implementations.

5. **Module-level singleton addiction** — 30+ module-level `let` variables create hidden global state. Two render loops in the same process corrupt each other. This is the single largest architectural debt.
